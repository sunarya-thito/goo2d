# Documentation Tasklist

Follow these strict rules for documenting the engine files:
1. **Concise Summary**: Start with a single sentence ending in a period.
2. **Two Paragraphs**: Non-trivial members (classes, mixins, constructors, non-private methods/fields) require at least two paragraphs explaining the purpose and usage.
3. **Natural Flow**: Do **NOT** use explicit "Why:" or "How:" labels. Integrate explanations naturally.
4. **Parameters**: Document each public parameter using `* [name]: explanation`.
5. **Generics**: Use `&lt;T&gt;` or `\<T\>` outside code blocks. No literal `<` or `>` allowed.
6. **Code Examples**: All public classes and libraries must include a ` ```dart ` code example.
7. **Cross-Links**: Classes must have a "See also:" section at the end with at least one `[Class]` link.
8. **No Placeholders**: Never use `TODO`, `FIXME`, or `...` (three dots) in documentation.
9. **No @nodoc**: All public members must be documented; `@nodoc` is forbidden.
10. **Overrides**: Do **NOT** document members with `@override`.
11. **Getters**: Do **NOT** include parameter documentation for getters.

**Common Code Example Mistakes to Avoid**:
- **Scope Mismatch**: Do not mix top-level declarations (like classes/mixins) with procedural code that belongs inside a `main()` or function. If the example shows usage, keep it procedural; if it shows implementation, keep it top-level.
- **Undeclared Symbols**: Ensure all variables, fields, and classes used in the example are either imported (like `Game`, `Component`) or declared within the snippet itself.
- **Three Dots**: Never use `...` as a placeholder in code blocks.
